# nano-claude 学习指南

一步步教你从零构建一个 AI 编程助手。每个版本只引入一个概念 —— 先 checkout 对应的 tag 看代码，再回来读"为什么"。

```bash
git checkout v0   # 从这里开始
```

---

## v0 — Chatbot 基线

**解决什么问题：** 在构建 Agent 之前，先理解一个纯聊天机器人长什么样 —— 以及它*做不了*什么。

**核心概念：** 聊天机器人就是一个循环：读取用户输入 → 发送给 API → 流式输出回复 → 循环。整个东西约 80 行代码，只有一个文件：

```
用户输入 → 调 API（流式）→ 逐 token 打印 → 循环
```

没有工具。模型能回答问题，但不能读文件、不能跑命令、不能修改代码。它被困在自己的上下文窗口里。

**阅读文件：**
- `src/index.ts` — 整个聊天机器人在一个文件里

**注意观察：**
- `messages` 数组累积对话历史 —— 这就是多轮对话的实现方式
- 流式使用 `client.messages.stream()` 处理 `content_block_delta` 事件
- 模型是*无状态的* —— 客户端在 messages 数组中维护所有状态

**动手试试：** 让聊天机器人读一个文件。它做不到 —— 要么道歉，要么胡编。这正是 v1 要解决的限制。

## v1 — Agent 工具调用循环

**解决什么问题：** 怎么让一个语言模型从"只会聊天"变成"能写代码、跑命令"？

**核心概念：** 整个 AI 编程助手的魔法浓缩在一个循环里：

```
用户输入 → 调 API → 模型返回文本或 tool_use → 执行工具 → 再调 API → ...
```

模型永远不会直接运行代码。它返回结构化的 `tool_use` 块，说"我想用参数 Y 调用工具 X"。你的代码执行工具、把结果喂回去、再问一次模型。循环往复，直到模型只返回文本（表示它做完了）。

**关键洞察：** 工具结果以 `user` 消息发送。这是 Claude API 的约定 —— 模型把工具结果当作用户提供的上下文，而不是自己的输出。所以消息历史里 `assistant`（tool_use）和 `user`（tool_result）是交替出现的。

**阅读文件：**
- `src/index.ts` — `runConversationLoop()` 是核心循环（约 30 行驱动一切）
- `src/api.ts` — `streamMessage()` 处理流式协议
- `src/tools/` — 每个工具都是简单的 `{ name, description, inputSchema, call() }` 对象

**6 个初始工具：**

| 工具 | 作用 | 模型为什么需要它 |
|------|------|-----------------|
| Bash | 运行 Shell 命令 | 安装依赖、跑测试、git 操作 |
| Read | 带行号读取文件 | 编辑前先理解代码 |
| Write | 创建/覆盖文件 | 生成新文件 |
| Edit | 文件中的字符串替换 | 精确修改已有代码 |
| Glob | 按模式搜索文件 | 在陌生代码库中导航 |
| Grep | 搜索文件内容 | 找到定义和引用位置 |

**动手试试：** checkout `v1`，从 `allTools` 中移除 Grep 工具，看模型如何适应 —— 它会改用 `Bash` 执行 `grep` 命令。这展示了模型的灵活性。

## v2 — 多 Provider 支持 + Markdown 渲染

**解决什么问题：** 不是所有人都直接用 Anthropic API。代理、网关、其他端点都需要兼容。而且终端里的纯文本输出太丑了。

**核心概念：API 适配器模式。** Anthropic SDK 接受 `baseURL` 参数，把所有请求重定向到不同端点。一个配置项就能兼容任何说 Anthropic API 协议的代理：

```typescript
// src/api.ts
new Anthropic({
  apiKey: config.apiKey || process.env.ANTHROPIC_API_KEY,
  baseURL: config.baseURL || process.env.ANTHROPIC_BASE_URL || undefined,
});
```

**Markdown 渲染**用两遍方式：先流式输出原始文本（即时反馈），再用 ANSI 转义码擦除、重新渲染带格式的版本。兼顾低延迟和美观输出。

**阅读文件：**
- `src/api.ts` — `createClient()` 的 baseURL 支持
- `src/render.ts` — 87 行的 Markdown 渲染器（代码块、标题、列表、行内格式）

**动手试试：** 把 `ANTHROPIC_BASE_URL` 设成一个无效 URL 观察报错。再指向一个可用的代理验证适配器模式。

## v3 — 权限确认 + Bash 安全分级

**解决什么问题：** 一个能不问就跑 `rm -rf /` 的 AI 就是一个定时炸弹。

**核心概念：两级风险分类。**

```
safe（安全）       → 自动放行（Read、Glob、Grep）
write（写入）      → 黄色警告，请求确认（git commit、npm install、Write、Edit）
destructive（危险） → 红色警告，请求确认（rm -rf、git reset --hard、DROP TABLE）
```

Bash 命令用正则模式匹配分级。"write"和"destructive"的区别在于可逆性：`rm file.txt` 可恢复（git、回收站、备份），但 `rm -rf /` 不可逆。

**阅读文件：**
- `src/permissions.ts` — `classifyBashRisk()` 有正则模式，`askPermission()` 处理交互确认

**设计决策：** 为什么不直接阻止所有危险命令？因为有时候用户*就是想* `rm -rf node_modules`。系统的角色是让用户**有意识地确认**，而不是阻止操作。

**动手试试：** 让模型执行 `git commit`，观察黄色确认提示。再试试让它执行 `rm -rf` —— 注意红色警告。

## v4 — 会话持久化 + 恢复

**解决什么问题：** 关掉终端就丢失整个对话。长时间编码会话不能接受这个。

**核心概念：** 每次交互后自动保存会话到 `~/.nano-claude/sessions/<id>.json`。会话文件包含完整消息历史、模型名、token 计数和元数据。可以列出、浏览、恢复会话。

**阅读文件：**
- `src/session.ts` — `saveSession()`、`loadSession()`、`listSessions()`
- `src/index.ts` — 每次 API 调用后自动保存，`/resume` 命令

**关键设计：** 会话保存的是*原始消息数组* —— 即 API 期望的格式。恢复时直接加载数组继续循环，无需重新解析或格式转换。

**动手试试：** 开始一段对话，关掉终端，重启后输入 `/sessions` 查看保存的会话。再用 `/resume <id>` 继续。

## v5 — 智能对话压缩

**解决什么问题：** Claude 有上下文窗口限制（~200K tokens）。包含大量工具调用的长编码会话最终会溢出。

**核心概念：** 不截断消息（会丢失上下文），而是用模型自身来总结对话。压缩算法：

1. 保留最后 2 条消息（最近的上下文）
2. 把其他所有内容发给模型请求"总结这段对话"
3. 用 `[摘要] + [助手确认] + [最后 2 条]` 替换旧消息

**为什么保留最后 2 条？** 它们包含当前用户请求和最新的助手回复 —— 当下正在发生什么的关键上下文。

**为什么选 75% 阈值？** 在上下文限制的 75% 处触发压缩，留出 25% 缓冲区。模型需要空间在压缩后生成回复 + 工具调用。太高 → 有中途截断的风险；太低 → 过早压缩浪费上下文空间。

**阅读文件：**
- `src/compact.ts` — `smartCompact()`、`shouldAutoCompact()`、`estimateTokens()`

**动手试试：** 用 `/cost` 观察 token 使用量增长。变大后手动执行 `/compact` 看压缩效果。

## v6 — 多层级 CLAUDE.md + 记忆

**解决什么问题：** 模型不知道你项目的规范、技术栈和偏好。每次对话都是冷启动。

**核心概念：通过 CLAUDE.md 文件实现项目感知。** 这些是多层级的 Markdown 文件，注入到系统 prompt 中：

```
~/.claude/CLAUDE.md              → 用户级默认值（你的名字、风格偏好）
<project-root>/CLAUDE.md         → 项目规则（技术栈、规范）
<project-root>/.claude/CLAUDE.md → 同上，备选位置
<cwd>/CLAUDE.md                  → 子目录级覆盖
```

所有层级都会加载并拼接。类似 `.gitignore` 的工作方式 —— 全局默认值加项目级覆盖。

**持久记忆**（`/remember`、`/forget`）将键值对存储为 `~/.nano-claude/memory/` 中的 Markdown 文件。每次 API 调用时都加载到系统 prompt，赋予模型跨会话记忆。

**阅读文件：**
- `src/context.ts` — `getClaudeMd()` 多层级加载和去重
- `src/memory.ts` — 简单的持久化键值记忆 CRUD
- `src/prompt.ts` — `buildSystemPrompt()` 组装所有内容

**动手试试：** 在项目根目录创建 `CLAUDE.md`，写入"始终使用 TypeScript。不要用 `any`。"然后让模型写代码 —— 它会遵循你的规则。

## v7 — 子 Agent 工具

**解决什么问题：** 有些任务需要读 10+ 个文件来收集上下文才能回答。在主对话里做这些会让上下文充满工具调用噪音。

**核心概念：** 子 Agent 是一个隔离的对话，只能读（不能写）。它有自己的消息历史，最多运行 20 轮工具调用，最后向主对话返回文本摘要。

**安全边界：** 子 Agent 只获得 `Read`、`Glob`、`Grep` —— 没有 `Bash`、`Write` 或 `Edit`。这防止了无人监管的修改。主对话中的模型可以通过子 Agent 请求研究，用户不需要批准每次文件读取。

**架构：**

```
主对话 ──[tool_use: Agent]──→ 子 Agent 对话
                                ├── Read 读取文件
                                ├── Grep 搜索模式
                                ├── Glob 查找文件
                                └── 返回摘要文本
      ←──[tool_result: 摘要]──┘
```

**阅读文件：**
- `src/tools/agent.ts` — 完整的子 Agent 实现

**动手试试：** 让模型"分析这个项目的架构"。看它生成子 Agent 读取多个文件，然后返回整合的摘要。

## v8 — CLI 参数 + 非交互模式 + 打磨

**解决什么问题：** 交互式 REPL 适合探索，但脚本和自动化需要非交互模式。同时，能用的原型不等于完成的项目 —— 文档、错误处理、边界情况都很重要。

**核心概念：** 无外部依赖的 CLI 参数解析，支持：

```bash
# 脚本：管道友好的输出
nano-claude -p "列出所有 TODO 注释" > todos.txt

# 自动化：恢复并继续
nano-claude -c -p "我们之前在做什么？"
```

**Print 模式的区别：**
- 无 Markdown 渲染（纯文本输出）
- 无 `[tool]`/`[result]` 噪音
- 自动允许所有工具权限（非交互）
- 模型完成后自动退出

**还改了什么：**
- 完整 README，包含架构图、工具表、与 Claude Code 的对比
- 所有工具的一致错误消息
- 边界情况处理（空文件、缺失目录、循环路径）

**阅读文件：**
- `src/index.ts` — `parseArgs()`，`runConversationLoop()` 中的 print 模式分支

## 架构总览

9 个版本（v0–v8）之后，完整架构是这样的：

```
┌─ 用户输入 ────────────────────────────────────────┐
│  CLI 参数 / REPL / 管道                           │
└──────────────┬────────────────────────────────────┘
               ▼
┌─ 系统 Prompt ─────────────────────────────────────┐
│  核心提示 + 环境 + git + CLAUDE.md + 记忆         │
└──────────────┬────────────────────────────────────┘
               ▼
┌─ Agent 循环 ──────────────────────────────────────┐
│  while (true) {                                   │
│    如需要则自动压缩                                │
│    流式调用 API                                   │
│    如果返回文本 → 渲染 Markdown，退出循环          │
│    如果返回 tool_use → 检查权限 → 执行             │
│    将 tool_result 作为 user 消息推入               │
│  }                                                │
└──────────────┬────────────────────────────────────┘
               ▼
┌─ 工具 ────────────────────────────────────────────┐
│  内置：Bash Read Write Edit Glob Grep Agent       │
└──────────────┬────────────────────────────────────┘
               ▼
┌─ 持久化 ──────────────────────────────────────────┐
│  会话   (~/.nano-claude/sessions/)                │
│  记忆   (~/.nano-claude/memory/)                  │
└───────────────────────────────────────────────────┘
```

## 核心收获

1. **模型就是 Agent。** 代码只是一个提供工具和管理上下文的外壳。智能来自 API。

2. **工具调用循环就是一切。** `while(true) { 调 API → 执行工具 → 重复 }` 就是整个架构。其他都是优化。

3. **上下文管理是核心难题。** 模型的表现取决于上下文窗口里放了什么。压缩、CLAUDE.md、记忆都是把正确上下文放到模型面前的策略。

4. **安全是关于确认，而非阻止。** 用户应该始终能做他们想做的事 —— 系统的职责是让危险操作成为*有意识的*选择。

5. **~1,900 行就够了。** Claude Code 有 50 万行，但核心 agent 循环只有约 30 行。其他都是 UI、边界情况和企业功能。理解这个核心就足以构建你自己的。
